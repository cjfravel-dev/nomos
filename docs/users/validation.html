Validation

Nomos embeds the template into the generated code, so every model can validate JSON at runtime — and you can extend validation with named, registered hooks.

Embedded validation

The generated NomosFormats object holds the reconstructed template and a validator built from it. Each model's validate checks JSON against that embedded schema, then parses — no template file is needed at runtime. 

import com.example.models.user.User

User.validate("""{"id":"1","username":"jo","email":"jo@x.com","roles":["admin"]}""") match {
  case Right(user)  => // a validated User
  case Left(errors) => errors.foreach(e => println(e.fullMessage))
}

Validation targets a definition by its fully-qualified name internally (e.g. com.example.models.user.User), so models in different sub-packages never collide.

Validation errors

Failures return a List[ValidationError]; each carries the JSON path, message, expected, and actual.

case class ValidationError(path: String, message: String, expected: String, actual: String) {
  def fullMessage: String  // "path: message (expected: …, got: …)"
}

Schema checks include: required vs. optional fields, type mismatches, string/number/array constraints, enum membership, whole-number checks for int/long, temporal parsing for date/datetime, discriminator resolution, and unexpected extra keys (unless $additionalProperties allows them).

Validating with the API

You can also validate without generated code by parsing a template directly.

import dev.cjfravel.nomos.Nomos

val template = Nomos.parseTemplate(templateJson, "com.example").right.get
Nomos.validate(template, jsonData, "com.example.User") match {
  case Right(_)      => // valid
  case Left(errors)  => // List[ValidationError]
}

Custom validators

Schema describes shape; anything beyond it is a custom validator — an arbitrary check that receives the whole JSON node and returns errors. Declare named validators on a definition, and register their implementations once at startup. Common uses include rules that span several fields or depend on one another (e.g. startDate <= endDate, or "if A then B is required"), but a validator can run any logic you like. 

{
  "name": "Reservation",
  "validators": ["dates.startBeforeEnd"],
  "template": { "startDate": "string", "endDate": "string" }
}
import dev.cjfravel.nomos.Nomos
import dev.cjfravel.nomos.validation.ValidationError

Nomos.validators.register("dates.startBeforeEnd") { json =>
  def str(name: String) = json.asObject.flatMap(_.field(name)).flatMap(_.asString)
  if (str("startDate").zip(str("endDate")).forall { case (s, e) => s <= e }) Nil
  else List(ValidationError("endDate", "must be >= startDate", "ordered", "unordered"))
}

Registered validators run after schema validation passes, and their errors are tagged with the validator name.

Custom string formats

The format constraint is backed by a registry. Built-ins are seeded; add your own domain formats.

Nomos.formats.register("hexColor")(_.matches("^#[0-9a-fA-F]{6}$"))
{ "color": { "type": "string", "format": "hexColor" } }

(De)serialization adapters

Per-field adapters convert between an on-the-wire form and your model's form — useful for legacy encodings.

{ "createdAt": { "type": "string", "adapter": "epochMillis" } }
Nomos.adapters.register("epochMillis")(
  decode = wire => wire,   // String => String
  encode = model => model
)
