JSON templating engine for Scala

One template is the law your data follows.

Nomos generates Scala case classes and a runtime JSON validator from a single declarative template — so your model and its validation are always derived from the same source of truth, and can never drift apart.

A hand-written model and its validator are two copies of the same truth, kept in sync by hand. Add a field, tighten a constraint, rename a variant — now you edit the case class, its (de)serialization, and its validation, and hope they agree. They drift. Nomos removes the second copy: you write one template, and the case class, its (de)serialization, and a validator are all generated from it.

In Greek, nomos (νόμος) is the law — the agreed-upon order a community lives by. Nomos makes your template the law: the single declaration your Scala types and your JSON both answer to.

Compared to annotation libraries

Most JSON tooling annotates a hand-written model — serialization hints and validation constraints declared as field annotations (Gson's @SerializedName, Bean Validation's @NotNull/@Size, and the like). The class stays the source of truth, and serialization and validation rules are scattered across annotations on its fields. Nomos inverts that: the template is the source of truth, and the class, its (de)serialization, and its validator are generated from it.

Annotating a model

  • You author and maintain the class by hand.
  • Rules are spread across field annotations; the full schema isn't readable in one place.
  • The schema is bound to one class in one language — JSON consumers elsewhere can't reuse it.
  • Custom validation beyond single fields means annotations plus a separate validator class, wired onto the model.
  • Reordering or renaming means touching the class, its serialization hints, and its constraints together.

Declaring with Nomos

  • One language-neutral template declares the shape; the class is generated.
  • Types, constraints, and structure live together in the declaration.
  • The same template can validate JSON, embed in the artifact, and serve as a shared contract.
  • Custom validators (any check over the payload, such as comparing or combining several fields) are named in the template and registered as functions; formats and adapters extend the same way.
  • Field order and naming come from one place and stay consistent across model, JSON, and validation.

The trade-off is a build-time generation step instead of plain hand-written classes — in return, the model and its validation are always derived from the same declaration.

A quick look

Drop a template under your resources, matching the package you want:

// src/main/resources/nomos/templates/com/example/models/user.json
{
  "definitions": [
    {
      "name": "User",
      "subPackage": "user",
      "template": {
        "id": "string",
        "username": { "type": "string", "minLength": 1 },
        "email": { "type": "string", "format": "email" },
        "age": { "$optional": "int" },
        "roles": ["string"]
      }
    }
  ]
}

Run mvn generate-sources, and Nomos writes a case class plus a companion with parse, serialize, and validate:

import com.example.models.user.User

// Validate against the embedded template, then parse — no template file needed at runtime
User.validate("""{"id":"1","username":"jo","email":"jo@x.com","roles":["admin"]}""") match {
  case Right(user)  => println(user.email)
  case Left(errors) => errors.foreach(e => println(e.fullMessage))
}

// Or round-trip directly
val json = User.toJson(user)
val back = User.fromJson(json)   // Either[String, User]

What you get

Rich type system

Strings, numbers (int, long, decimal, double), date/datetime, booleans, arrays, maps, unions, enums, defaults, and nullable raw fields.

Validation built in

The template is embedded in the generated code, so every model validates JSON at runtime with no separate schema to ship or keep in sync.

Polymorphic families

Type discriminators become sealed traits with variants — including $ref bodies, prefix-matched tags, and shared variant collapse.

Extensible

Register named custom validators, custom string formats, and (de)serialization adapters via Nomos.validators, Nomos.formats, and Nomos.adapters.

Build-time generation

A Maven plugin scans your templates and generates sources during generate-sources. Output to src/main/scala or a gitignored folder.

Multi-file projects

Split a large model one template per package; $ref resolves across files into a shared definition space.
