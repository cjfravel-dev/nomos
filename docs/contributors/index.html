Contributing to Nomos

How to set up a dev environment, build and test the modules, and land a change.

Prerequisites

  • JDK 11+ — the build targets Java 8 bytecode via --release 8. CI verifies the build on Java 11 and Java 17.
  • Maven 3.9+
  • Scala 2.12.21 — managed by the build; no separate install needed.

Build & test

# build, test, verify, and install all reactor modules
mvn clean install -Dgpg.skip=true

# rebuild the example end-to-end against the locally installed artifacts
mvn -f nomos-example/pom.xml clean test

The nomos-example module is the end-to-end proof: the plugin regenerates real models from templates, then a test suite round-trips JSON through the generated code.

Coverage

Statement coverage for nomos-runtime and nomos-core runs via scoverage:

mvn -pl nomos-runtime,nomos-core scoverage:report scoverage:check
# HTML reports: nomos-runtime/target/site/scoverage/index.html
#               nomos-core/target/site/scoverage/index.html

CI enforces the measured runtime baseline at 45% and the core floor at 80%.

Test-driven workflow

Changes are developed test-first: add a failing spec under nomos-core/src/test/scala/dev/cjfravel/nomos/, implement until it passes, then prove anything user-visible end-to-end in nomos-example. A feature that changes generated code should have both a content test (what's emitted) and an example test (that it compiles and runs).

Docs & release

User docs live under docs/ and are served as the project site. Two build-time gates keep them honest: dev/scripts/readme-has-version.sh fails the build unless release docs and every consumer POM reference the current version, and dev/scripts/docs-no-stale-refs.sh fails it if any doc or comment names a third-party JSON library or a removed symbol. Regenerate the API reference under docs/api and docs/runtime-api with dev/scripts/build-api-docs.sh whenever the public Scala surface changes. CI regenerates both references and fails if the checked-in output differs.

Releases publish to Maven Central via mvn deploy. Publishing stages a droppable deployment that requires a manual Publish in the Central portal (autoPublish is off), and CI verifies the release artifacts.

Code style

  • Wrap if/else branch bodies in braces.
  • Comments describe present behavior — not history, removed code, or session notes.
  • Prefer order-preserving ListMap where field/variant order matters.

Pull requests

  • Keep changes focused; update the relevant docs page when behavior changes.
  • Ensure mvn -pl nomos-core test and the example build are green.
  • Describe the user-visible effect of the change.
