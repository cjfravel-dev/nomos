Releasing Nomos

A published GitHub Release builds, signs, publishes, and verifies the matching Maven Central version.

Prepare the version

  1. Update the reactor, example, README, docs, and changelog to one non-SNAPSHOT version.
  2. Regenerate API documentation with dev/scripts/build-api-docs.sh.
  3. Merge the focused release pull request after Java 11/17 CI and CodeQL pass.

Publish the GitHub Release

  1. Create a tag exactly equal to the Maven project version, without a v prefix.
  2. Review release notes and mark prerelease versions appropriately.
  3. Publish the GitHub Release. This is the intentional release approval.

The protected Maven Central workflow checks the tag/version pair, credential expiration, signing-key fingerprint, and duplicate-version status. It then runs the reactor and example, signs every Central artifact, publishes only after Central validation succeeds, waits for the published state, and tests a clean consumer.

Safe defaults

A local mvn deploy uses user-managed Central deployment by default and stops after validation. Only the protected GitHub Release workflow enables automatic publication.

Failure handling

  • A build, test, signing, upload, or Central validation failure publishes nothing.
  • A version already present on Central fails before deployment.
  • A published Central version is immutable; corrections require a new version.
  • Rotate the Central token and PGP key before the expiration dates recorded in the maven-central environment.
